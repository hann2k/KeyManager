## 완료

- **1단계 — 로컬 Named Pipe 브로커** (`KeyManager.App`). 항목별 AES-256-GCM 금고, Argon2id, §9 프로토콜, 클라이언트 SDK, 트레이 관리 UI, 다국어.
- **2단계 — TCP/TLS 분리** (`KeyManager.Server` + `KeyManager.MasterGui` 비상주). C1 봉투 모델(zero-knowledge 서버), 금고/봉투 송수신 3종, TOFU 핀 TLS, admin 토큰 인증, 서버 포트 설정, 기존 로컬 금고 이관.
  - ⚠️ **현재 서버는 loopback(127.0.0.1) 바인딩 = 사실상 로컬 전용.** 원격 접속은 아래 "TCP 개방"에서.
- **서버 코어 분리 (토대 리팩터, Windows 완료)** — 서버 런타임을 WinForms 껍데기에서 떼어 공유 라이브러리 `KeyManager.ServerCore`(Protocol만 참조 → zero-knowledge 유지)로 이동. `KeyManager.Server`는 **헤드리스 콘솔/서비스 호스트**(Generic Host + `UseWindowsService()`, `BackgroundService`가 TCP/TLS 구동, 최초 실행 시 admin 토큰 콘솔 출력, Ctrl+C/SCM 정지 시 우아한 종료)로 전환. 트레이 UI는 별도 동반 앱 `KeyManager.Tray`로 분리(같은 폴더의 서버 기동·감시, 상태 표시, `StopSignal` 명명 이벤트로 우아한 정지). Server·Tray는 같은 폴더로 배포(`publish.ps1`).
  - 아직 Windows 서비스로 **설치되진 않음**(호스트 리팩터만). Linux 데몬/컨테이너 호스트는 아래 "다중플랫폼"에서.

## 다음 A — TCP 개방(hardened): IP 기반 운영을 안전하게

서버를 IP로 열기 전 **반드시** 갖출 것(우선순위 순). 근거: 지금 열면 비밀은 안 새지만(zero-knowledge) 열거·DoS·읽기 무인증이 노출됨.

1. **원격 바인딩 옵션** — `IPAddress.Loopback` → `IPAddress.Any`(+ 인터페이스/방화벽 선택). 기본은 loopback 유지, 명시적으로 켤 때만 개방.
2. **읽기 인증 닫기 (설계 결정 2)** — 가장 큰 한 방. 마스터 GUI가 push 때 클라이언트별 검증키 `K_fetch = HMAC(S, "fetchauth")`를 봉투에 함께 올리고, 소비앱은 `HMAC(K_fetch, nonce‖timeStep)`로 응답, 서버가 저장한 `K_fetch`로 대조. → `S` 없는 자의 **열람·열거·오프라인 확보 차단**. 서버는 여전히 `S`를 모르니 zero-knowledge 유지.
3. **가용성 방어** — 동시 핸들러 상한(`SemaphoreSlim`), IP별 rate limit(토큰버킷), 핸드셰이크/읽기 타임아웃, in-flight 상한. 상시 서버엔 필수.
4. **전송 신뢰 강화** — 순수 TOFU의 첫-접속 MITM 갭 보완: 기기 프로비저닝 때 서버 인증서 지문 out-of-band 주입, 또는 사설 CA / **mTLS(기기별 클라 인증서)**. mTLS면 전송 계층에서 기기 인증(=식별)까지 해결.
5. **감사/로깅** — (2번 적용 후) 인증된 fetch 로그 + 이상 패턴 알림.

> **배치 권장:** 인터넷 직노출 대신 **LAN + VPN(WireGuard/Tailscale)**. 공격면을 "전 인터넷" → "오버레이 네트워크 참여자"로 축소하고, 위 1~4는 심층 방어로 얹는다.

## 다음 B — 다중 관리자(쓰기) 동시성

현재 `pushVault`는 금고 전체 교체 = **last-write-wins**, 시작 후 재동기화 없음 → 동시 편집 시 조용한 유실. 전제: **단일 금고 / 단일 마스터 암호 / 단일 admin 토큰**.

- **편집 리스(single-writer lock)** — 서버가 한 번에 한 명에게만 편집권 리스 발급, 나머지는 읽기전용/대기. *한 팀이 같은 금고를 나눠 관리*하는 모델에 최적, 구현 작음. **1순위 후보.**
- **낙관적 동시성(version/etag)** — 금고에 `version` 필드, push 시 서버가 최신과 대조·불일치면 거부 → GUI가 재pull·재적용 후 재시도. 진짜 동시 편집이 필요할 때.
- (선택) debounce 자동 push → **명시적 "서버 저장"** 으로 전환(clobber 창 축소).

> 지금 당장 안전한 사용: **동시 편집자 1명**(순차 사용은 시작 시 fresh pull이라 OK).

## 다음 C — 신원·자격 모델 (등록은 사람+UI)

**원칙: 데이터 평면은 자동/직통, 관리 평면은 사람+UI.**
- **런타임 fetch(데이터 평면)** = 기계가 자동으로 서버와 직통, 자격으로. ← 자동화 OK(그게 목적).
- **등록·부여(관리 평면)** = 사람이 UI 앞에서. **프로그램적 클라이언트 자동 등록 없음**(열리면 자가 등록 = 최악의 공격면). 웹으로 가더라도 "프런트만 교체", 등록은 여전히 **사람 + UI + 승인**.
  - ("등록을 서비스로 분리"는 UI가 호출하는 **내부 계층**을 뜻함 — 클라가 직접 때리는 공개 엔드포인트가 아님.)

**자격(개인 식별부호) 진화 방향:**
- 지금: 서버가 생성해 건네는 **대칭 시드 `S`**(베어러 열쇠 — 신원과 자격이 안 나뉨).
- 지향: **클라이언트가 로컬에서 키쌍 생성 → 공개키를 식별부호로 등록**(비밀키는 절대 밖으로 안 나감). 런타임 인증 = challenge에 서명 → 서버가 공개키로 검증. 봉투는 공개키로 봉인(ECIES/하이브리드) → 관리자·서버·GUI 아무도 소비앱 비밀을 안 만짐.
- 이 전환이 **"다음 A"의 읽기 인증(결정 2)을 자연히 겸함**(자격을 서버가 실제 검증). 등록은 사람이 UI에서 공개키를 붙여넣는 행위라 자동 API 불필요.

**셀프서비스 ≠ 셀프부여:** 웹 관리 UI로 확장해 클라이언트 측 사람이 직접 요청하더라도, 어떤 키를 받을지는 **관리자 승인 게이트** 필수(OAuth dynamic registration + approval 패턴).

## 다중플랫폼 (웹은 필수 관문 아님)

"다중플랫폼"은 세 평면으로 갈리고 답이 각각 다름:
- **소비앱(데이터 평면): 이미 완료.** `Client`/`Protocol`이 net10.0(portable) → Linux/Mac/컨테이너/WSL에서 이미 fetch 가능.
- **서버: 헤드리스 크로스플랫폼 호스트.** 스마트홈이면 Pi/NAS/Docker 상시 구동이 핵심 → **웹이 아니라** 서버 코어를 WinForms 껍데기에서 분리해 플랫폼별 호스트(Windows=트레이, Linux=데몬/컨테이너).
  - ✅ **Windows 토대 완료:** 서버 코어를 `KeyManager.ServerCore` 라이브러리로 떼고, `KeyManager.Server`를 헤드리스 콘솔/서비스 호스트로, 트레이를 별도 동반 앱(`KeyManager.Tray`)으로 분리(위 "완료" 참조). 남은 것: **Linux 데몬/컨테이너 호스트**(`ServerCore`가 아직 net10.0-windows이므로 아래 "토대 리팩터"의 portable화가 선행) + Windows 서비스 설치(`sc.exe`/installer).
- **관리 UI: 여기서만 "웹"이 거론됨.** 크로스플랫폼 네이티브(**Avalonia / .NET MAUI**)가 `Kd`를 관리자 기기에 두는 보안 모델을 유지하므로 우선. **웹은 모바일/무설치가 꼭 필요할 때만**, 그때도 `Kd` 위치 문제(관리자 로컬이 웹을 서빙하는 변형)를 같이 풀어야 함.

**토대 리팩터(선행):** 서버 코어를 WinForms 껍데기에서 분리하는 **1차 토대(server core → `ServerCore` lib + 헤드리스 호스트 + 트레이 동반 앱)는 Windows에서 완료**(위 "완료" 참조). 남은 portable화 과제: `KeyManager.Core`/`KeyManager.ServerCore`가 net10.0-**windows**인 유일한 이유는 1단계 Named Pipe(ACL/PID) 및 WinForms 의존. Windows 전용 조각을 전용 어셈블리로 떼고 **코어를 net10.0(portable)로** → 헤드리스 서버(Linux 데몬/컨테이너)·크로스플랫폼 관리 UI 양쪽이 그 위에 얹힘(UI 선택과 무관).

**우선순위:** ① 코어 portable화(남은 토대) → ② 헤드리스 크로스플랫폼 서버 = Linux 데몬/컨테이너(Windows 헤드리스 호스트는 이미 완료, 스마트홈 최대 이득) → ③ 크로스플랫폼 관리 UI(필요 시, 웹보다 Avalonia/MAUI 우선). **웹 전환은 목적지 선택일 뿐 필수 단계 아님.**

## 나중 — 멀티테넌트 (관리자별 독립 금고)

관리자마다 **자기 암호로 자기 금고**. 서버가 금고를 신원별로 N개 보관·라우팅, 관리자별 토큰. "남의 금고라 못 연다" 문제 원천 제거. 서버 저장 포맷·인증·라우팅 재설계 필요 = 실질적 새 기능. **다중 관리자가 서로 독립적으로 쓸 때** 필요.

## 유스케이스 드라이버 — 스마트홈 키 서버

IP 기반 상시 키 서버로 기기들에 키 배급. 개방이 전제이므로 **"다음 A" 1~4가 사실상 진입 요건.**
- **safety-critical**(현관 잠금 등): 다음 A 1~4 필수 + 크립토 경로 외부 리뷰 권장.
- **편의 키**(날씨/일반 API 등): 개방 + rate limit로 충분.

## 백로그 (미정)

- **프로세스/기기 바인딩 부활** — 1단계의 Named Pipe PID 바인딩(§13)은 TCP엔 없음 → mTLS 기기 인증으로 대체 검토. `ClientRecord.BoundProcessPath`는 현재 미사용.
- **소비앱 쓰기(저장) 요청** — 별도 권한 체크 포함(설계 §15 "2단계 이후").
- **키 백업/복구 + 복구 코드(recovery key)**.
