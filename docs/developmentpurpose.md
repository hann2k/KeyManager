# KeyManager — Design Notes (pre-spec review)

> Status: **discussion / design stage.** Whether to start building was decided after reviewing this.
> The original notes are preserved at the bottom under [Appendix].
> (Korean version: [개발목적.md](개발목적.md))

---

## 1. One-line definition

> A **local secret broker** (tray app) that keeps API keys / secret data **encrypted at rest** and,
> when an **authorized local app** asks, hands back that value safely.
> (Optional) the cloud can be added as a zero-knowledge backend that **only syncs the encrypted vault**.

The key differentiator is not mere storage but the **"central broker that distributes keys to many local apps"** behavior.
Only the **local agent** can decrypt; neither the consumer apps nor the cloud ever hold plaintext.

---

## 2. Terminology — there are two kinds of "client" (watch out)

In this document "client" refers to different things depending on context.

| Context | "client" | "server" |
|---|---|---|
| **Cloud sync** | **local agent** (= this app; holds the master key, en/decrypts) | cloud (stores ciphertext only) |
| **Local brokering** | **consumer app** (another program that receives keys; doesn't know the master key) | local agent |

- **Local agent** = the tray app we build. The only thing that can decrypt.
- **Consumer app** = an external program that asks the agent for keys. Receives only values within its scope.
- Decryption-key custody = the **local agent's** job. Consumer apps and the cloud never hold the key.

---

## 3. Tech stack (decided)

| Item | Decision |
|---|---|
| Language/runtime | **C# / .NET 10** (pure C#, no C++ DLL) |
| GUI | WinForms, lives in the tray → double-click for the UI |
| Local IPC | **Named Pipe** by default (§7). TCP loopback is an optional adapter |
| Crypto | AES-256-GCM (at rest & transit), Argon2id (key derivation), HMAC-SHA256 (auth) |

- **Why no C++:** plaintext values have to reach the GUI anyway (for visual confirmation on entry), so a C++ core's memory-hygiene benefit disappears — only the interop cost remains.
- **Transport is abstracted (`ISecretTransport`).** Named pipe today; swap in a TLS/HTTP adapter later if cloud/remote is needed — the core logic stays put.
- The core store is also isolated behind `ISecretStore` so it can be swapped.

---

## 4. Two kinds of key — never mix them (the backbone of the design)

| | **(1) at-rest key `Kd`** | **(2) auth/transport key (time-based)** |
|---|---|---|
| Role | encrypt values on disk | prove consumer identity + protect value in transit |
| Stability | **fixed (immutable)** | **changes constantly** |
| Source | master password → KDF | per-client shared seed `S` + time/nonce |
| Why | changing it forces re-encrypting everything | must change to defeat eavesdropping/replay |

> Time-based values are used **only for (2) auth/transport.** Never put time into the at-rest key.

---

## 5. The at-rest key is regenerated from a "formula (KDF)", not stored as text

```
Stored on disk :  salt + Argon2 parameters   (the key itself is never stored)
On unlock      :  Kd = Argon2id(master password, salt)   ← regenerated by formula
On lock/idle   :  Kd wiped from memory
```

- The decryption key does not exist as text on disk.
- It is reproduced via the "formula" only when the master password is supplied.
- **The master key (`Kd`) is never sent to the cloud** — the lifeline of zero-knowledge (§9).

---

## 6. Data model

### 6.1 Secret vault — values are always ciphertext
```
vault file (syncable to cloud as-is — all ciphertext)
 ├─ entry : { id, name(encrypted), value(encrypted), createdAt, updatedAt }
 ├─ entry : ...
 └─ ...
```
- **Per-entry encryption** (not one monolithic blob) → only the requested entry needs decrypting (§8, minimize residency).
- **Names are encrypted too** → prevents leaking metadata ("which service keys does this person hold") from disk/cloud.
- The GUI is unlocked, so it can decrypt names with `Kd` to render the list.

### 6.2 Consumer (client) registry — OAuth client+scope model
```
{ clientName,
  seed S (random, shown only once at registration → set in the consumer app),
  allowedKeys [k1, k3, ...],          ← keys this app may access
  boundProcessPath / signature (optional)  ← executable binding (stronger identity)
}
```
- Seed `S` is also stored encrypted with `Kd`. Decrypted only at verification time → discarded after use.

---

## 7. Local IPC mechanism — Named Pipe (not TCP)

Since it's the same machine, **Named Pipe** beats TCP loopback on almost every axis.

| | TCP 127.0.0.1 | **Named Pipe** ✅ |
|---|---|---|
| Access control | ❌ any local process can connect | ✅ built-in ACL (restrict to my account/SID) |
| Client PID | awkward (`GetExtendedTcpTable`) | ✅ `GetNamedPipeClientProcessId`, one line |
| Network exposure | goes through the network stack | ✅ unreachable from the network |
| Port/firewall | port clashes, prompts | ✅ just a name |
| Eavesdropping | possible (npcap, etc.) | hard without kernel-level access |
| Client ubiquity | best | supported in .NET/C++/Python (slightly less universal) |

- **Decision: Named Pipe by default.** TCP loopback only as an optional adapter (`ISecretTransport`) when needed.
- Because the Named Pipe ACL blocks "who can connect" at the OS level, the transport encryption in §9 drops to **defense-in-depth (secondary)** — but the structure is kept.

---

## 8. Memory residency policy (reflects dump-defense intent)

| Target | Resident? |
|---|---|
| decrypted **values** | ❌ not resident. Only the one requested item is decrypted at the moment → response → wiped immediately |
| client seed `S` | ❌ decrypted only at verification → wiped |
| at-rest key `Kd` | ⚠️ resident only while unlocked (unavoidable). **Auto-wiped on idle timeout** |
| locked state | every request rejected |

> Caveat (honest): C#'s `string` can't be reliably zeroed. Exposure "at the instant of processing" can't be fully prevented. The goal is to **minimize residency** — keeping plaintext out of disk/swap/idle dumps.

---

## 9. Local protocol (over Named Pipe)

The original `[request key][secret key]` sketch, extended with **authorization + replay defense**.

```
[connect]  agent: issue a random challenge (nonce)         ← fully blocks replay
[request]  consumer: authCode = HMAC(S, "auth" ‖ nonce ‖ timeStep)
                    → to agent  {clientName, requestKey, nonce, authCode}
[verify]   agent: ① recompute authCode, check time window (±1 step)
                  ② check nonce unused (seen-nonce cache)
                  ③ GetNamedPipeClientProcessId → path/signature check (optional)
                  ④ requestKey ∈ this client's allowedKeys?
[serve]    agent: decrypt only that one entry with Kd → plaintext
                  sessionKey = HMAC(S, "enc" ‖ nonce)        ← derived from S directly (never sent)
                  ciphertext = AES-GCM(sessionKey, plaintext) sent → plaintext wiped at once
[client]   consumer: sessionKey = HMAC(S, "enc" ‖ nonce)     ← derives the same from its own S
                  AES-GCM-decrypt and use the value
```

### Key derivation — the only real secret is `S`
- **`S` (the shared seed) never travels over the wire.** Set in the consumer app once, at registration.
- `authCode` and `sessionKey` are both computed **independently on each side** from `S` + public values (nonce/timeStep) → no need to send them.
- **The encryption key (`sessionKey`) is never transmitted.** "Encrypt with the code you received" is forbidden — that code is sent in the clear, so an eavesdropper could decrypt with it.
- **Domain-separation labels** (`"auth"` / `"enc"`) split the access token from the lock key → seeing `authCode` reveals nothing about `sessionKey`.

| Travels the wire (OK to expose) | Never travels |
|---|---|
| `nonce`, `timeStep`, `authCode`, ciphertext | `S`, `sessionKey`, plaintext values, `Kd` |

- **Time-based** (timeStep): adds freshness. Being local, server and client share the same system clock → no skew.
- **Server nonce (challenge)**: blocks replay even within the time window.
- Message framing is length-prefixed + JSON Lines (avoids `:`/`]` breakage).

### Supported requests
| Request | Response |
|---|---|
| `getKey(clientName, key, nonce, authCode)` | encrypted `value` (via sessionKey) |
| `listKeys(clientName, nonce, authCode)` | the client's `allowedKeys` names |

## 10. Write/read paths (the zero-knowledge flow)

Plaintext exists only **inside the agent**, briefly (at entry, and at the moment of decryption).

### Write (registering a key) — by the user, in the GUI
```
① user types a new key into the agent GUI  → the only moment plaintext is visible (visual confirmation)
② agent encrypts with Kd                    → name/value = ciphertext
③ append the entry to the vault file
④ (optional) register/sync the vault (all ciphertext) to the cloud
```

### Read (querying a key) — by a consumer over IPC
```
① (optional) agent downloads the ciphertext vault from cloud (whole — safe, it's ciphertext)
② consumer requests via the §9 protocol
③ agent decrypts only that one entry with Kd → re-encrypts with sessionKey → consumer decrypts
```

- Consumer apps **don't participate in writes.** (App-initiated storage is a future option, with its own permission check.)
- Only **ciphertext** ever flows to/from the cloud.

### 10.1 Encryption is two layers — the consumer only removes ②
| Layer | Who removes it | Key |
|---|---|---|
| **① at-rest** (inside the vault) | the agent only | `Kd` (master key) |
| **② transport** (on the wire) | only the requesting consumer | `sessionKey` (derived from `S`) |

```
vault [① Kd] ─agent strips ①→ plaintext ─wraps ② sessionKey→ wire [② enc]
                                                       │ consumer computes sessionKey from its own S
                                                       ▼ strips ② → obtains plaintext
```
- The consumer **never touches `Kd`.** It computes `sessionKey` from its own `S` and removes **only the transport layer (②)**.
- This ②-decryption is handled by the **consumer-side client SDK** → the app developer writes one line, `km.Get("openai")`. (This is what the original note's "the client must be able to decrypt too" really meant = transport decryption.)

### 10.2 "If the consumer ends up with plaintext anyway, what's the point of encryption?"
- It isn't a leak, it's **delivery** — the app actually needs the plaintext to call the API.
- Encryption protects **everything except that delivery**: disk/cloud (ciphertext only), unauthorized apps, the GUI list (names only), the master key (never leaves the agent).
- In short: "**an authorized app sees only permitted keys, only at the moment of use.**" Every other path is blocked. That is the broker's purpose.

## 11. Cloud (optional) — zero-knowledge hybrid

### Trade-off: you can only have two of three
```
        ① cloud hosting
            /        \
  ② consumers need no key    ③ only I can decrypt (server can't read)
```
- ①+② (server decrypts and distributes): a cloud breach leaks everything → violates the original "only I can decrypt".
- ①+③ (server holds only ciphertext): safe, but the consumer needs the master key → loses the broker's advantage.
- ②+③: the current local design (no cloud).

### Keep all three with a hybrid (the Bitwarden/1Password way)
```
[cloud]  stores/syncs only the encrypted vault. No decryption key (zero-knowledge)
     │  syncs while encrypted
     ▼
[local agent on each PC]  decrypts locally with the master key. Brokers to local apps over named pipe
     │  named pipe
     ▼
[consumer apps]  ask without knowing the key, and receive
```
- **The decrypting broker is always local.** The cloud exists only as an "unreadable vault" → ①②③ all preserved.
- The cloud's only value = **syncing the encrypted vault across PCs**.
- Why downloading the whole thing is safe = not "because it's small" but **because it's ciphertext**. It even leaks less access-pattern metadata than per-key requests.

### When you might not need to build this at all
- If clients are on **another machine / serverless** where no local agent can run → you'd have to accept ①+② (server decrypts), which is essentially re-implementing **HashiCorp Vault / AWS Secrets Manager / Infisical** → consider the off-the-shelf option.

## 12. GUI behavior

- Lives in the tray → shows the window on double-click.
- List: shows **key names only** (values hidden).
- A value is shown in plaintext **only at the instant of entry** (to eyeball what you just typed). No re-view in the GUI after saving (policy in §15).
- A screen to register consumer apps and manage their permissions (allowedKeys).
- Lock/unlock and idle auto-lock.

---

## 13. Stronger client identity (a Named Pipe bonus)

Stronger than letting the consumer **self-declare** its name:
- On a Named Pipe connection → get the PID via `GetNamedPipeClientProcessId` (no race) → resolve the executable path via `QueryFullProcessImageName` → compare against the registry's `boundProcessPath` (+ code signature).
- Blocks a malicious process from merely spoofing a name. (Optional hardening.)

## 14. Logic-check results

| # | Check | Result |
|---|---|---|
| 1 | "master key non-resident" vs "decrypt on request" | ⚠️ **contradiction → resolved.** Consumers don't know the master password, so `Kd` must be resident while unlocked. "Non-resident" applies only to **values**. On lock, wipe `Kd` + reject requests |
| 2 | Use a time-based key for at-rest encryption? | ❌ forbidden. Time-based is auth/transport only (§4) |
| 3 | Store the at-rest key as text? | ❌ derive via KDF each time, store only the salt (§5) |
| 4 | Replay defense from time-based alone? | △ insufficient → reinforced with a server nonce (challenge) (§9) |
| 5 | "Encrypt with the received code"? | ❌ dangerous → sessionKey is never sent; both sides derive from S, with domain separation (§9) |
| 6 | Metadata leak from storing names in plaintext | solved by encrypting names too (§6.1) |
| 7 | Is syncing the whole vault to cloud a violation? | ✅ safe — it's ciphertext. But keep per-entry encryption inside (§6.1, §11) |
| 8 | A single blob would prevent single-entry decryption | solved by per-entry encryption → whole-sync and single decryption coexist (§6.1) |
| 9 | TCP loopback lacks access control | replaced by Named Pipe + ACL (§7) |
| 10 | Integrity/tampering | AES-GCM detects per-entry tampering. (A file-level MAC for rollback/deletion detection is optional) |
| 11 | Headless service right after boot | not possible — no service before unlock. Rejecting requests is the correct behavior |

### Remaining intrinsic limits (honestly)
- The shared seed `S` must be **stored on the consumer side.** An attacker who fully owns the same user account can read that store and forge codes.
- With account takeover, a process memory dump can also expose `Kd` — **the same would be true in C++.**
- If the master password is lost, even the cloud can't recover it (the cost of zero-knowledge).
- These defenses **raise the bar significantly** but don't 100% stop "full takeover of the same account" → a pragmatic best effort.

## 15. Phase 1 (MVP) scope — decided

> **Cloud is excluded from phase 1.** Finish the local-only broker first. Transport is abstracted via `ISecretTransport` so cloud can be layered on later as an adapter.

Included:
- [x] Per-entry encrypted vault (single JSON file), names + values encrypted
- [x] Argon2id master-key derivation + AES-256-GCM
- [x] Named Pipe broker + §9 protocol (`getKey`/`listKeys`)
- [x] Consumer registry (`S`, allowedKeys) + client SDK + sample consumer app
- [x] WinForms tray + unlock/lock + key/client management UI

Decided defaults:
- Idle auto-lock: **disabled** (consumers fetch keys frequently). Manual lock only. Re-enable just by setting `AutoLockAfter`.
- DPAPI auto-unlock: **disabled** (security first, opt-in only)
- Value re-view: **only once, at entry** (re-view after master re-confirmation to be considered later)
- Process binding (§13): **optional** in phase 1 (off by default), scaffolding in place

Phase 2 and beyond (undecided):
- [ ] Cloud sync backend (self-hosted vs S3-compatible) and conflict/version handling
- [ ] Key backup/recovery + a recovery key
- [ ] Write (store) requests initiated by consumer apps
