# KeyManager — 설계 정리 (스펙 확정 전 검토본)

> 상태: **대화/설계 단계.** 개발 착수 여부는 이 문서 검토 후 결정.
> 원본 메모는 문서 맨 아래 [부록] 보존.
> (English version: [developmentpurpose.md](developmentpurpose.md))

---

## 1. 한 줄 정의

> 로컬에 상주하면서 API 키·비밀 데이터를 **암호화 저장**하고,
> **인가된 로컬 앱**이 질의하면 해당 값을 안전하게 배급하는 **로컬 시크릿 브로커** (트레이 앱).
> (선택) 클라우드는 **암호문 금고를 동기화만** 하는 zero-knowledge 백엔드로 확장 가능.

핵심 차별점은 단순 저장이 아니라 **"여러 로컬 앱에 키를 중앙 배급하는 브로커"** 동작이다.
복호화 능력은 **로컬 에이전트에만** 있고, 소비 앱도 클라우드도 평문을 갖지 않는다.

---

## 2. 용어 — "클라이언트"가 두 종류다 (혼동 주의)

이 문서에서 "클라이언트"는 맥락에 따라 다른 대상을 가리킨다.

| 맥락 | "클라이언트" | "서버" |
|---|---|---|
| **클라우드 sync** | **로컬 에이전트** (= 이 앱. 마스터 키 보유, 암·복호화 수행) | 클라우드 (암호문만 보관) |
| **로컬 브로커링** | **소비 앱** (키를 받아 쓰는 다른 프로그램. 마스터 키 모름) | 로컬 에이전트 |

- **로컬 에이전트** = 우리가 만드는 트레이 앱. 유일하게 복호화 능력을 가진다.
- **소비 앱** = 에이전트에 키를 물어보는 외부 프로그램. 권한 범위 내 값만 받는다.
- 복호화 키 관리 = **로컬 에이전트** 몫. 소비 앱·클라우드는 절대 키를 갖지 않는다.

## 3. 기술 스택 (확정)

| 항목 | 결정 |
|---|---|
| 언어/런타임 | **C# / .NET 10** (순수 C#, C++ DLL 안 씀) |
| GUI | WinForms, 트레이 상주 → 더블클릭 시 UI |
| 로컬 IPC | **Named Pipe** 기본 (§7). TCP loopback은 선택적 어댑터 |
| 암호 | AES-256-GCM(저장·전송), Argon2id(키 유도), HMAC-SHA256(인증) |

- **C++ 안 쓰는 이유:** 평문 값이 어차피 GUI로 와야 하므로(입력 시 육안 확인) C++ 코어의 메모리 위생 이점이 사라진다. 인터롭 비용만 남음.
- **전송은 추상화한다(`ISecretTransport`).** 오늘은 named pipe, 훗날 클라우드/원격이 필요하면 TLS·HTTP 어댑터를 갈아끼운다 — 코어 로직은 그대로.
- 코어 저장소도 `ISecretStore`로 분리해 교체 가능하게 둔다.

## 4. 키는 두 종류 — 절대 섞지 않는다 (설계의 뼈대)

| | **(1) 저장 암호화 키 `Kd`** | **(2) 인증/전송 키 (시간 기반)** |
|---|---|---|
| 역할 | 디스크의 값 암호화 | 클라(소비 앱) 신원 증명 + 전송 중 값 보호 |
| 안정성 | **고정 (불변)** | **수시로 변함** |
| 출처 | 마스터 암호 → KDF | 클라별 공유 시드 `S` + 시간/nonce |
| 이유 | 바뀌면 전체 재암호화 필요 | 바뀌어야 도청·재전송 무력화 |

> 시간 기반 값은 **(2) 인증/전송에만** 쓴다. 저장 암호화 키에 시간을 넣지 않는다.

---

## 5. 저장 키는 "텍스트"가 아니라 "수식(KDF)"으로 그때그때 생성

```
디스크에 저장하는 것 :  salt + Argon2 파라미터   (키 자체는 절대 저장 안 함)
unlock 시점          :  Kd = Argon2id(마스터암호, salt)     ← 수식으로 생성
lock/유휴 시점       :  Kd 메모리에서 소거
```

- 복호화 키는 디스크에 텍스트로 존재하지 않는다.
- 마스터 암호가 들어올 때만 "수식"으로 재현된다.
- **마스터 키(`Kd`)는 클라우드로 절대 전송하지 않는다** — zero-knowledge의 생명줄(§9).

## 6. 데이터 모델

### 6.1 시크릿 금고 (vault) — 값은 항상 암호문
```
vault 파일 (통째로 클라우드 sync 가능 — 전부 암호문)
 ├─ entry : { id, name(암호화), value(암호화), createdAt, updatedAt }
 ├─ entry : ...
 └─ ...
```
- **항목별 개별 암호화**한다 (단일 덩어리 블롭 아님) → 요청 1건만 복호화 가능(§8 상주 최소화).
- **이름도 암호화**한다 → 디스크/클라우드에서 "어떤 서비스 키를 보유 중인지"(메타데이터) 누출 방지.
- GUI는 unlock 상태이므로 `Kd`로 이름을 복호화해 목록 표시 가능.

### 6.2 클라이언트(소비 앱) 등록표 — OAuth client+scope 모델
```
{ clientName,
  seed S (랜덤, 등록 시 1회만 표시 → 소비 앱에 설정),
  allowedKeys [k1, k3, ...],          ← 이 앱이 접근 가능한 키 목록
  boundProcessPath / signature (선택)  ← 실행파일 바인딩(신원 강화)
}
```
- 시드 `S`도 `Kd`로 암호화해 저장. 검증 시에만 복호화 → 사용 후 폐기.

## 7. 로컬 IPC 방식 — Named Pipe (TCP 아님)

같은 머신이므로 TCP loopback보다 **Named Pipe**가 거의 모든 면에서 낫다.

| | TCP 127.0.0.1 | **Named Pipe** ✅ |
|---|---|---|
| 접근 통제 | ❌ 아무 로컬 프로세스나 접속 | ✅ ACL 내장(내 계정/SID 제한) |
| 클라 PID 확인 | 까다로움(`GetExtendedTcpTable`) | ✅ `GetNamedPipeClientProcessId` 한 줄 |
| 네트워크 노출 | 네트워크 스택 경유 | ✅ 네트워크에서 도달 불가 |
| 포트/방화벽 | 포트 충돌·팝업 | ✅ 이름만 |
| 도청 | npcap 등 가능 | 커널 레벨 아니면 어려움 |
| 클라 범용성 | 최고 | .NET/C++/Python 등 지원(약간 덜 보편) |

- **결정: Named Pipe 기본.** TCP loopback은 필요 시 켜는 선택적 어댑터(`ISecretTransport`)로만.
- Named Pipe ACL이 "누가 접속하나"를 OS 차원에서 막아주므로, §9의 전송 암호화는 **심층 방어(부차)** 수준으로 내려간다. 그래도 구조는 유지.

---

## 8. 메모리 상주 정책 (덤프 방어 의도 반영)

| 대상 | 상주 여부 |
|---|---|
| 복호화된 **값들** | ❌ 상주 안 함. 요청 1건만 그 순간 복호화 → 응답 → 즉시 폐기 |
| 클라 시드 `S` | ❌ 검증 시에만 복호화 → 폐기 |
| 저장 키 `Kd` | ⚠️ unlock~lock 동안만 상주(불가피). **유휴 타임아웃 시 자동 소거** |
| 잠금 상태 | 모든 요청 거부 |

> 주의(솔직): C#의 `string`은 확실한 0-소거가 어렵다. "처리되는 그 순간"의 노출은 완전히 못 막는다. 목표는 **상주 시간 최소화** — 디스크/스왑/유휴 덤프에 평문이 남지 않게 하는 수준.

## 9. 로컬 프로토콜 (Named Pipe 위)

원본의 `[request key][secret key]` 단순안을 **인가·재전송 방어**가 들어간 형태로 확장.

```
[연결]  에이전트: 랜덤 challenge(nonce) 발급            ← 재전송 완전 차단용
[요청]  소비앱: authCode = HMAC(S, "auth" ‖ nonce ‖ timeStep)
                → 에이전트로  {clientName, requestKey, nonce, authCode}
[검증]  에이전트: ① authCode 재계산·시간창(±1 step) 확인
                  ② nonce 미사용 확인(seen-nonce 캐시)
                  ③ GetNamedPipeClientProcessId → 경로/서명 확인(선택)
                  ④ requestKey ∈ 해당 클라 allowedKeys?
[처리]  에이전트: 해당 1건만 Kd로 복호화 → 평문
                  sessionKey = HMAC(S, "enc" ‖ nonce)        ← S로 직접 유도 (전송 안 함)
                  암호문 = AES-GCM(sessionKey, 평문) 전송 → 평문 즉시 폐기
[클라]  소비앱: sessionKey = HMAC(S, "enc" ‖ nonce)           ← 자기 S로 동일하게 유도
                AES-GCM 복호화하여 값 사용
```

### 키 유도 — 진짜 비밀은 `S` 하나뿐
- **`S`(공유 시드)는 절대 와이어로 흐르지 않는다.** 등록 시 1회만 소비 앱에 설정.
- `authCode`·`sessionKey`는 둘 다 `S` + 공개값(nonce/timeStep)에서 **양쪽이 각자 계산** → 와이어로 보낼 필요 없음.
- **암호화 키(`sessionKey`)는 전송하지 않는다.** "받은 코드로 암호화"는 금지 — 그 코드는 전송돼 도청자가 그대로 복호화 가능.
- **도메인 분리 라벨**(`"auth"` / `"enc"`)로 출입증과 자물쇠 열쇠를 갈라놓는다 → `authCode`를 봐도 `sessionKey` 추론 불가.

| 와이어로 흐름 (노출 OK) | 절대 안 흐름 |
|---|---|
| `nonce`, `timeStep`, `authCode`, 암호문 | `S`, `sessionKey`, 값 평문, `Kd` |

- **시간 기반**(timeStep): 코드에 신선도 부여. 로컬이라 서버·클라 같은 시스템 시계 → 시계 오차 없음.
- **server nonce(challenge)**: 시간창 내 재전송까지 원천 차단.
- 메시지 포맷은 길이 프리픽스 + JSON Lines (`:`/`]` 깨짐 방지).

### 지원 요청
| 요청 | 응답 |
|---|---|
| `getKey(clientName, key, nonce, authCode)` | 암호화된 `value` (sessionKey로) |
| `listKeys(clientName, nonce, authCode)` | 해당 클라의 `allowedKeys` 이름 목록 |

## 10. 쓰기/읽기 경로 (zero-knowledge 흐름)

평문은 **에이전트 안에서만**(입력 순간·복호화 순간) 잠깐 존재한다.

### 쓰기 (키 등록) — 사용자가 GUI에서
```
① 사용자가 에이전트 GUI에 새 키 입력  → 평문이 보이는 유일한 순간(육안 확인)
② 에이전트가 Kd로 암호화            → name/value = 암호문
③ vault 파일에 항목 추가
④ (선택) vault(전부 암호문)를 클라우드에 등록/sync
```

### 읽기 (키 조회) — 소비 앱이 IPC로
```
① (선택) 에이전트가 클라우드에서 암호문 vault 내려받음(통째 OK — 암호문이라 안전)
② 소비 앱이 §9 프로토콜로 요청
③ 에이전트가 요청 1건만 Kd로 복호화 → sessionKey로 재암호화 전송 → 소비 앱이 복호화
```

- 소비 앱은 **쓰기에 관여하지 않는다.** (앱이 직접 키를 저장하는 기능은 향후 선택 사항이며, 그때 별도 권한 체크가 붙는다.)
- 클라우드엔 **항상 암호문만** 오간다.

### 10.1 암호화는 두 겹 — 소비 앱은 ②만 푼다
| 겹 | 푸는 주체 | 키 |
|---|---|---|
| **① 저장 암호** (vault 안) | 에이전트만 | `Kd` (마스터 키) |
| **② 전송 암호** (와이어) | 요청한 소비 앱만 | `sessionKey` (= `S`에서 유도) |

```
vault [① Kd 암호] ─에이전트가 ① 벗김→ 평문 ─② sessionKey 씌움→ 와이어 [② 암호]
                                                          │ 소비 앱이 자기 S로 sessionKey 계산
                                                          ▼ ② 벗김 → 평문 획득
```
- 소비 앱은 **`Kd`를 절대 안 만진다.** 자기 `S`로 `sessionKey`를 계산해 **전송 암호(②)만** 푼다.
- 이 ②-복호화는 **소비 앱용 클라이언트 SDK**가 처리 → 앱 개발자는 `km.Get("openai")` 한 줄. (원본 메모의 "클라이언트도 복호화 가능해야"의 정체 = 전송 복호화)

### 10.2 "소비 앱이 결국 평문을 가지면 암호화 의미 없지 않나?"
- 아니다. 그건 누출이 아니라 **배달**이다 — 그 앱은 실제로 그 키로 API를 호출해야 하므로 평문이 필요.
- 암호화가 지키는 건 **배달을 뺀 전부**: 디스크/클라우드(①암호문만), 권한 없는 앱, GUI 목록(이름만), 마스터 키(에이전트 밖으로 안 나감).
- 즉 "**인가된 앱이 허가된 키만, 사용하는 순간에만** 평문을 본다." 나머지 경로는 모두 차단. 그것이 브로커의 목적.

## 11. 클라우드(선택) — zero-knowledge 하이브리드

### 트레이드오프: 셋 중 둘만 가능
```
        ① 클라우드 호스팅
            /        \
  ② 소비앱은 키 불필요    ③ 나만 복호화(서버도 못 읽음)
```
- ①+②(서버가 복호화 배급): 클라우드 침해 시 전부 유출 → 원본 요구 "나만 복호화" 위반.
- ①+③(서버는 암호문만): 안전하지만 클라가 마스터 키 필요 → 브로커 장점 상실.
- ②+③: 지금의 로컬 설계(클라우드 없음).

### 하이브리드로 셋 다 살린다 (Bitwarden/1Password 방식)
```
[클라우드]  암호문 금고만 보관/동기화. 복호화 키 없음(zero-knowledge)
     │  암호화된 채로 sync
     ▼
[각 PC의 로컬 에이전트]  마스터 키로 로컬 복호화. 로컬 앱엔 named pipe로 브로커링
     │  named pipe
     ▼
[소비 앱들]  키 몰라도 물어보면 받음
```
- **복호화 브로커는 항상 로컬.** 클라우드는 "못 읽는 금고"로만 존재 → ①②③ 전부 유지.
- 클라우드가 주는 가치 = **여러 PC 간 암호화 금고 동기화**뿐.
- 통째로 받아도 안전한 이유 = "양이 적어서"가 아니라 **암호문이라서**. 오히려 키 단위 요청보다 접근-패턴 메타데이터 누출이 적다.

### 직접 만들 필요가 없을 수도 있는 경우
- 클라이언트가 **다른 머신/서버리스**라 로컬 에이전트를 못 두면 → ①+②(서버 복호화)를 받아들여야 하고, 이는 사실상 **HashiCorp Vault / AWS Secrets Manager / Infisical** 재구현 → 기성품 검토 권장.

- 트레이 상주 → 더블클릭 시 창 표시.
- 목록: **키 이름만** 표시(값 비노출).
- 값은 **입력하는 그 순간에만** 평문 표시(방금 친 값 육안 확인용). 저장 후 GUI 재열람 불가(정책은 §15).
- 소비 앱 등록/권한(allowedKeys) 관리 화면.
- 잠금/해제(unlock) 및 유휴 자동 잠금.

## 13. 클라이언트 신원 강화 (Named Pipe의 보너스)

소비 앱이 이름을 **자기 신고**하는 것보다 강하게:
- Named Pipe 연결 → `GetNamedPipeClientProcessId`로 PID 확보(레이스 없음) → `QueryFullProcessImageName`으로 실행파일 경로 확인 → 등록표 `boundProcessPath`와 대조(+코드 서명).
- 악성 프로세스가 이름만 사칭하는 것을 차단. (선택적 강화 옵션)

## 14. 논리 검증 결과

| # | 점검 | 결과 |
|---|---|---|
| 1 | "마스터 키 비상주" vs "요청 시 복호화" | ⚠️ **모순 → 해결.** 소비 앱은 마스터 암호를 모르므로 `Kd`는 unlock 동안 상주 불가피. "비상주"는 **값**에만 적용. 잠금 시 `Kd` 소거 + 요청 거부 |
| 2 | 시간 기반 키를 저장 암호화에 사용? | ❌ 금지. 시간 기반은 인증/전송 전용(§4) |
| 3 | 저장 키를 텍스트로 보관? | ❌ KDF로 그때그때 유도, salt만 저장(§5) |
| 4 | 시간 기반만으로 재전송 방어? | △ 불충분 → server nonce(challenge) 보완(§9) |
| 5 | "받은 코드로 암호화"? | ❌ 위험 → sessionKey는 전송 안 하고 양쪽이 S로 유도, 도메인 분리(§9) |
| 6 | 이름 평문 저장 시 메타데이터 누출 | 이름도 암호화로 해결(§6.1) |
| 7 | 클라우드에 vault 통째 전송 위반? | ✅ 안전. 암호문이라 가능. 단 vault 내부는 항목별 암호화 유지(§6.1, §11) |
| 8 | 단일 블롭 암호화면 1건 복호화 불가 | 항목별 암호화로 해결 → 통째 sync와 1건 복호화 동시 성립(§6.1) |
| 9 | TCP loopback 접근 통제 부재 | Named Pipe + ACL로 대체(§7) |
| 10 | 무결성/변조 | AES-GCM 항목별 변조 탐지. (롤백/삭제 탐지용 파일 MAC은 선택) |
| 11 | 부팅 직후 헤드리스 서비스 | 불가 — unlock 전엔 서비스 불가. 요청 거부가 정상 |

### 남는 본질적 한계 (정직하게)
- 공유 시드 `S`는 **소비 앱 측이 저장**해야 한다. 같은 사용자 계정을 완전 장악한 공격자는 그 저장소를 읽어 코드를 위조할 수 있다.
- 같은 계정 장악 시 프로세스 메모리 덤프로 `Kd`도 노출 가능 — **C++이었어도 동일**.
- 마스터 암호 분실 시 클라우드도 복구 불가(zero-knowledge의 대가).
- 이 방어들은 **공격 난도를 크게 올릴 뿐**, "같은 계정 완전 장악"을 100% 막지는 못한다 → 현실적 최선.

## 15. 1단계(MVP) 범위 — 확정

> **클라우드는 1단계에서 제외.** 로컬 전용 브로커를 먼저 완성한다. 전송은 `ISecretTransport`로 추상화해 클라우드는 나중에 어댑터로 얹는다.

포함:
- [x] 항목별 암호화 vault(단일 JSON 파일), 이름·값 암호화
- [x] Argon2id 마스터 키 유도 + AES-256-GCM
- [x] Named Pipe 브로커 + §9 프로토콜(`getKey`/`listKeys`)
- [x] 소비 앱 등록표(`S`, allowedKeys) + 클라이언트 SDK + 샘플 소비 앱
- [x] WinForms 트레이 + unlock/잠금 + 키/클라이언트 관리 UI

확정된 기본값:
- 유휴 자동 잠금: **비활성**(소비 앱이 수시로 키를 꺼내 써야 하므로). 수동 잠금만. `AutoLockAfter` 값만 주면 부활 가능
- DPAPI 자동 unlock: **비활성**(보안 우선, 옵션으로만)
- 값 재열람: **입력 시 1회만**(추후 마스터 재확인 시 허용 검토)
- 프로세스 바인딩(§13): 1단계는 **선택**(기본 끔), 골격만 마련

2단계 이후:
- [~] **TCP 버전(진행 중)** — TCP/TLS로 상주 서버(`KeyManager.Server`) + 비상주 관리 GUI(`KeyManager.MasterGui`) 분리, C1 봉투 모델(zero-knowledge 서버). 구현 계약: [tcp-architecture.md](tcp-architecture.md).
- [ ] 클라우드 sync 백엔드(자체 서버 vs S3 호환) 및 충돌/버전 관리
- [ ] 키 백업/복구 + 복구 코드(recovery key)
- [ ] 소비 앱의 쓰기(저장) 요청 기능
